# CI/CD Release Pipeline + README Deployment Docs Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Extend `.github/workflows/release.yml` so a single release-please tag publishes the monorepo gh-pages panel, the all-in-one `metacubexd-server` multi-arch Docker image, and unsigned Electron desktop installers for all 6 OS/arch targets — then document every deployment mode in `README.md`.

**Architecture:** This plan is pure config + docs (no application code). The release workflow keeps its existing `release-please`, gh-pages, dist tarball, screenshots, and the legacy static `metacubexd` image jobs, but repoints build paths at `packages/ui` / `apps/server`, adds a new `release-server-image` job pointing at `apps/server/Dockerfile`, and adds a `release-desktop` matrix job that builds + uploads electron-builder artifacts. README gains hosted-panel, desktop-per-OS, docker-compose, profiles, custom-kernel, monorepo-dev, and troubleshooting sections.

**Tech Stack:** GitHub Actions (release-please, docker/build-push-action, softprops/action-gh-release, peaceiris/actions-gh-pages), `actionlint` for validation, pnpm 10 workspace, electron-builder 26, mihomo `v1.19.27`, Markdown.

**Depends on:** Plan 01 (monorepo migration → `packages/ui`, `apps/server`, `apps/desktop` exist with the exact package names below). Plan 03 (`apps/server/Dockerfile`) and Plan 04 (`apps/desktop/electron-builder.yml`, `apps/desktop/scripts/fetch-mihomo.mjs`) produce the files this workflow invokes; this plan's CI references those exact paths so the jobs are correct once those files land. Full end-to-end verification (a real GH release with all artifacts) requires those plans to be merged and a real tag pushed — this plan verifies via `actionlint` + a documented job-graph dry-run.

## Shared facts this plan relies on (from SHARED CONTRACTS + spec §7/§8/§10)

- Package names: root `metacubexd-monorepo` (private); UI `@metacubexd/ui`; agent `@metacubexd/agent`; desktop `@metacubexd/desktop`; server `@metacubexd/server`.
- UI publish build is `pnpm --filter @metacubexd/ui generate` → output at `packages/ui/.output/public`.
- Server Docker image: `apps/server/Dockerfile`, platforms `linux/amd64,linux/arm64`, build-arg `MIHOMO_VERSION`, tags `ghcr.io/metacubex/metacubexd-server:{latest,<tag>}`.

- Desktop pre-fetch script: `apps/desktop/scripts/fetch-mihomo.mjs` (provided by Plan 04; uses `@metacubexd/agent` `fetchKernel`/`mihomoAsset`).
- Desktop renderer copy target: `apps/desktop/renderer/` (the UI `.output/public` static export).
- electron-builder output dir: `apps/desktop/dist` (per spec §4 `directories.output: dist`).
- **Canonical control healthcheck path is `/api/control/health`** (SHARED CONTRACTS reconcile the spec — the README troubleshooting/docker sections MUST use `/api/control/health`, NOT the spec §5 Dockerfile's `/control/health`). The Dockerfile itself is owned by Plan 03; this plan only references the path in docs.
- Hosted panel CNAME stays `d.metacubex.one` (unchanged from current workflow).
- **Explicit choice (per prompt):** the OLD static-only `ghcr.io/metacubex/metacubexd` image is superseded by the gh-pages hosted panel, but we KEEP both the gh-pages job AND the legacy `release-image` job (it does not conflict with the new server image — different image name, different tags) and ADD the new `release-server-image` job. This is documented inline in the workflow with a comment.

### Task 3: Add the unsigned `release-desktop` electron matrix job

**Files:** Modify `.github/workflows/release.yml` (add job `release-desktop` after `release-server-image`, before/after `update-screenshots` — order does not matter as jobs are parallel).

**Why:** Spec §7(b) + §10 step 9. Six unsigned desktop installers (mac arm64 + x64, win x64 + arm64, linux x64 + arm64) built via electron-builder and attached to the release-please tag. The matrix splits mac across two runners (Apple Silicon `macos-14` for arm64, Intel `macos-13` for x64) and builds both arches on a single Windows/Linux runner. Each leg: `pnpm install` → UI `generate` (with `NUXT_APP_BASE_URL='./'`) → copy renderer → pre-fetch the per-arch mihomo → `electron-vite build` → `electron-builder <ebargs> -c.mac.identity=null --publish never` (with `CSC_IDENTITY_AUTO_DISCOVERY=false`) → upload files-only to the release.

- [ ] **Step 1: Add the `release-desktop` job.** Insert this job (after the `release-server-image` job from Task 2). Use it EXACTLY as written — it matches spec §7's matrix and the SHARED CONTRACTS paths:

  ```yaml
  release-desktop:
    needs: release-please
    if: ${{ needs.release-please.outputs.release_created }}

    permissions:
      contents: write

    strategy:
      fail-fast: false
      matrix:
        include:
          - { os: macos-14, ebargs: '--mac --arm64' }
          - { os: macos-13, ebargs: '--mac --x64' }
          - { os: windows-latest, ebargs: '--win --x64 --arm64' }
          - { os: ubuntu-latest, ebargs: '--linux --x64 --arm64' }

    runs-on: ${{ matrix.os }}

    env:
      MIHOMO_VERSION: v1.19.27
      CSC_IDENTITY_AUTO_DISCOVERY: 'false'

    steps:
      - uses: actions/checkout@v6

      - uses: pnpm/action-setup@v6

      - uses: actions/setup-node@v6
        with:
          cache: pnpm
          node-version: latest

      - name: install dependencies
        run: pnpm install

      - name: build ui (static, relative base for file://)
        run: pnpm --filter @metacubexd/ui generate
        env:
          NUXT_APP_BASE_URL: './'

      - name: copy ui into desktop renderer
        run: rm -rf apps/desktop/renderer && cp -r packages/ui/.output/public apps/desktop/renderer
        shell: bash

      - name: prefetch bundled mihomo (per arch)
        run: node apps/desktop/scripts/fetch-mihomo.mjs

      - name: electron-vite build (main + preload)
        run: pnpm --filter @metacubexd/desktop build

      - name: electron-builder (unsigned)
        run: pnpm --filter @metacubexd/desktop exec electron-builder ${{ matrix.ebargs }} -c.mac.identity=null --publish never
        shell: bash
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - name: attach desktop installers to release
        uses: softprops/action-gh-release@v3
        with:
          tag_name: ${{ needs.release-please.outputs.tag_name }}
          files: |
            apps/desktop/dist/*.dmg
            apps/desktop/dist/*.zip
            apps/desktop/dist/*.exe
            apps/desktop/dist/*.AppImage
            apps/desktop/dist/*.deb
  ```

  ````markdown
  ### 3. All-in-One Server (Docker)

  The `metacubexd-server` image bundles the dashboard UI, the control agent, and
  a per-arch mihomo kernel. One container serves the panel, supervises the
  kernel, and exposes the proxy.

  ```yaml
  # compose.yaml — proxy-only by default; TUN is an advanced override
  services:
    metacubexd:
      image: ghcr.io/metacubex/metacubexd-server:latest
      restart: unless-stopped
      environment:
        CONTROL_TOKEN: 'change-me-control'
        CLASH_SECRET: 'change-me-clash'
        CONTROL_PORT: '8080'
        CLASH_API_PORT: '9090'
        MIXED_PORT: '7890'
        TZ: 'Asia/Shanghai'
      ports:
        - '8080:8080' # dashboard UI + /api/control agent API
        - '9090:9090' # mihomo Clash API + WebSocket (UI endpoint target)
        - '7890:7890' # mixed proxy port
      volumes:
        - 'metacubexd-data:/data'

  volumes:
    metacubexd-data: {}
  ```

  ```shell
  docker compose up -d

  # Update
  docker compose pull && docker compose up -d
  ```

  Open `http://<host>:8080` for the dashboard. The control agent unlocks the
  kernel/profile UI automatically (it probes `/api/control/info` on the same
  origin).

  API directly (never proxied), so set the endpoint to:

  | Field  | Value                            |
  | :----- | :------------------------------- |
  | URL    | `http://<host>:9090`             |
  | Secret | the `CLASH_SECRET` you set above |

  #### Environment variables

  | Variable         | Default               | Purpose                                                                                                                             |
  | :--------------- | :-------------------- | :---------------------------------------------------------------------------------------------------------------------------------- |
  | `CONTROL_TOKEN`  | _(none)_              | Bearer token guarding the control agent (`/api/control/**`); also accepted as `?token=` for the SSE log stream. Set a strong value. |
  | `CLASH_SECRET`   | _(none)_              | Secret for mihomo's Clash API (`external-controller`). Use this as the UI endpoint's **Secret**.                                    |
  | `CONTROL_PORT`   | `8080`                | Port serving the dashboard UI + control agent API.                                                                                  |
  | `CLASH_API_PORT` | `9090`                | Port for mihomo's Clash API + WebSocket. The UI endpoint targets this port.                                                         |
  | `MIXED_PORT`     | `7890`                | mihomo mixed (HTTP + SOCKS) proxy port.                                                                                             |
  | `TZ`             | _(container default)_ | Timezone for logs/scheduling, e.g. `Asia/Shanghai`.                                                                                 |

  The named volume mounts `/data`, which holds your profiles, the active config,
  and the kernel's geo / fake-ip caches. It **must be writable** — a read-only
  data dir makes the kernel exit non-zero.

  #### TUN mode (advanced)

  TUN needs `NET_ADMIN`, the `/dev/net/tun` device, and host networking. Override
  the service:

  ```yaml

    metacubexd:
      image: ghcr.io/metacubex/metacubexd-server:latest
      restart: unless-stopped
      network_mode: host
      cap_add:
        - NET_ADMIN
      devices:
        - '/dev/net/tun:/dev/net/tun'
      environment:
        CONTROL_TOKEN: 'change-me-control'
        CLASH_SECRET: 'change-me-clash'
      volumes:
        - 'metacubexd-data:/data'

  volumes:
    metacubexd-data: {}
  ```

  With `network_mode: host` the `ports:` mapping is ignored — the container
  binds `8080`/`9090`/`7890` directly on the host. Enable a `tun:` block in your
  profile's mihomo config for the tunnel to come up.
  ````
